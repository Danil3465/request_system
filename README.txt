СИСТЕМА УЧЁТА ЗАЯВОК


1. О проекте

Проект представляет собой систему для учёта заявок сотрудников с контролем бизнес-правил.

Реализовано:
- REST API на FastAPI
- База данных PostgreSQL
- Автоматические миграции через Alembic
- Бизнес-логика с проверкой статусов
- Веб-интерфейс для работы с заявками


2. Функциональные возможности

Бэкенд:
- POST /requests - создание заявки
- GET /requests/{id} - просмотр заявки по ID
- GET /requests - список с фильтрацией по статусу, исполнителю, подразделению, просрочке
- PATCH /requests/{id}/status - изменение статуса с проверкой правил
- PATCH /requests/{id}/executor - изменение исполнителя
- GET /requests/report - отчёт по статусам, просрочкам, исполнителям

Фронтенд:
- Создание заявки
- Блок статистики
- Поиск заявки по ID с карточкой
- Изменение статуса у найденной заявки
- Фильтрация списка заявок
- Адаптивный дизайн

4. Структура базы данных

База данных состоит из четырёх таблиц:

Таблица employees - сотрудники.
Поля: id, full_name, job_title, department_id.

Таблица departments - подразделения.
Поля: id, name.

Таблица statuses - статусы заявок.
Поля: id, name.

Таблица requests - заявки.
Поля: id, created_at, author_id, executor_id, description, due_date, status_id.

Связи:
- author_id и executor_id ссылаются на employees.id
- status_id ссылается на statuses.id
- department_id ссылается на departments.id

Нормализация:
Статусы и подразделения вынесены в отдельные таблицы, чтобы исключить дублирование и упростить изменения.


5. Бизнес-правила

Порядок статусов:
Новая -> В работе -> Выполнена

Правила переходов:
- Из «Новой» можно только в «В работе»
- Из «В работе» можно только в «Выполнена»
- «Выполненную» заявку нельзя изменить
- Нельзя вернуть статус назад
- Нельзя перепрыгивать через статус

Все переходы проверяются в бизнес-логике.


6. Оптимизация производительности

Для запроса, который выбирает просроченные заявки конкретного исполнителя в статусе «В работе», создан составной индекс:

CREATE INDEX idx_requests_performance ON requests (executor_id, status_id, due_date);

Результаты замеров:

До оптимизации:
- Время выполнения: 547.941 ms
- Тип сканирования: Parallel Seq Scan

После оптимизации:
- Время выполнения: 2.922 ms
- Тип сканирования: Bitmap Index Scan

Ускорение: в 187 раз.
